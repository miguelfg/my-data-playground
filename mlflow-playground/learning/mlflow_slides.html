1. What is MLflow and what are its core components?

Answer:

MLflow is an open-source platform for managing the end-to-end machine learning lifecycle, including experiment tracking, model registry, deployment, and tracing capabilities for both traditional ML and LLM/GenAI applications.

Core Components:

  • MLflow Tracking: Logs parameters, metrics, code versions, and artifacts
  • MLflow Models: Standard format for packaging ML models with metadata
  • MLflow Model Registry: Centralized store for managing model lifecycle
  • MLflow Projects: Packaging format for reproducible runs
  • MLflow Tracing: Observability for LLM/agent applications
Source: MLflow Documentation

2. How do you log parameters, metrics, and artifacts?

Answer:

import mlflow

with mlflow.start_run():
    # Log parameters (inputs)
    mlflow.log_param("learning_rate", 0.01)
    mlflow.log_params({"epochs": 100, "batch_size": 32})
    
    # Log metrics (outputs)
    mlflow.log_metric("accuracy", 0.95)
    mlflow.log_metric("loss", 0.05, step=10)
    
    # Log artifacts (files)
    mlflow.log_artifact("model.pkl")
    mlflow.log_artifacts("outputs/")  # directory

Key Distinction:

  • Parameters: Inputs (hyperparameters)
  • Metrics: Numerical outputs tracked over time
  • Artifacts: Files (models, plots, data)
Source: MLflow Tracking API

4. Experiment vs Run in MLflow

Answer:

  • Experiment: Organizational unit grouping related runs (e.g., "Customer Churn Model v2")
  • Run: Single execution of model code with specific parameters, metrics, and artifacts

Think of it: Experiments are folders, runs are individual files.

mlflow.set_experiment("fraud-detection")  # Set/create experiment

with mlflow.start_run(run_name="baseline-model"):  # Create run
    # Training code here
    pass

One experiment contains many runs for comparison and analysis.

Source: MLflow Tracking Concepts

5. MLflow Model Registry & Stages

Answer:

Model Registry is a centralized model store with APIs and UI for managing the model lifecycle from experimentation to production.

Model Stages:

  • None: Newly registered
  • Staging: Being tested/validated
  • Production: Deployed and serving predictions
  • Archived: Deprecated but retained
# Register model
mlflow.register_model("runs:/abc123/model", "FraudDetector")

# Transition to production
client = mlflow.MlflowClient()
client.transition_model_version_stage(
    name="FraudDetector", 
    version=3, 
    stage="Production"
)
Source: MLflow Model Registry

6. What are MLflow model flavors?

Answer:

Model flavors are standardized interfaces for different frameworks and deployment targets.

Types of Flavors:

  • Framework-specific: sklearn, pytorch, tensorflow, xgboost
  • Universal: python_function (pyfunc) - works everywhere
  • Specialized: spark, onnx, transformers
# Save with multiple flavors
mlflow.sklearn.log_model(
    model, 
    "model",
    signature=signature,
    registered_model_name="MyModel"
)
# Automatically includes sklearn AND pyfunc flavors

Why it matters: Pyfunc provides a consistent predict() interface regardless of underlying framework, enabling framework-agnostic deployment.

Source: MLflow Models

7. Loading models for inference

Answer:

Multiple approaches depending on use case:

# Method 1: Load by run ID
model = mlflow.pyfunc.load_model("runs:/abc123/model")

# Method 2: Load from Model Registry
model = mlflow.pyfunc.load_model("models:/FraudDetector/Production")

# Method 3: Load with specific flavor
model = mlflow.sklearn.load_model("runs:/abc123/model")

# Make predictions
predictions = model.predict(new_data)

Best practice: Use Model Registry paths (models:/) for production systems as they abstract away run IDs and enable version management.

Source: MLflow Models Guide

8. Model signatures - why important?

Answer:

A model signature defines the input/output schema (column names, data types) for a model.

from mlflow.models import infer_signature

signature = infer_signature(X_train, model.predict(X_train))
mlflow.sklearn.log_model(model, "model", signature=signature)

Why Critical:

  • Validation: Catches data schema mismatches at serving time
  • Documentation: Self-documents expected inputs/outputs
  • Deployment: Required by many serving platforms
  • Type safety: Prevents silent errors from incorrect dtypes
Source: Model Signatures

9. Setting up MLflow Tracking Server

Basic (local development):

mlflow server --host 0.0.0.0 --port 5000

Production setup:

mlflow server \
    --backend-store-uri postgresql://user:pass@host:5432/mlflow \
    --default-artifact-root s3://my-bucket/mlflow-artifacts \
    --host 0.0.0.0 \
    --port 5000

Components:

  • Backend store: Metadata (PostgreSQL/MySQL for production)
  • Artifact store: Model files (S3, GCS, Azure Blob, NFS)
  • Frontend: UI and REST API

Production tip: Use managed services or containerize with authentication/HTTPS.

Source: MLflow Tracking Server

11. Comparing multiple runs

UI Method:

  1. Navigate to experiment in MLflow UI
  2. Select multiple runs (checkboxes)
  3. Click "Compare" button
  4. View parallel coordinates plot, metrics comparison, parameter diffs

Programmatic Method:

from mlflow.tracking import MlflowClient

client = MlflowClient()
runs = client.search_runs(
    experiment_ids=["1"],
    filter_string="metrics.accuracy > 0.9",
    order_by=["metrics.accuracy DESC"]
)

for run in runs:
    print(f"{run.info.run_id}: {run.data.metrics['accuracy']}")

Best practice: Tag runs meaningfully and use consistent naming for easy filtering.
